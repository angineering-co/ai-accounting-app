# 客戶端模組化導覽（Portal Navigation）
>
> **緣起**：PR #225（「其他文件」上傳 + `/documents`）合併後，portal 出現「首頁 / 憑證上傳中心」與「其他文件」兩個上傳目的地、語意相牴觸。深入後發現這是更大問題的早期症狀：portal 缺乏模組結構，而我們即將把 **報表 (IS/BS)** 加進 portal，後續還有 **營所稅**、**節稅** 模組。本文件先定方向，導覽改名只是其第一個實作切片。

## 1. 底層形狀：三層，而非四個並列模組

四個概念不是兄弟，分屬不同層：

- **第 1 層 — 輸入（客戶提供的資料）**：統一發票、折讓單、其他文件，以及（未來）營所稅的客戶填寫表單。
- **第 2 層 — 記帳核心（事務所內部產出）**：擷取 → 分錄（journal entries）→ 分類帳。對客戶大致隱形。
- **第 3 層 — 模組（核心之上的視圖 / 申報 / 洞察）**：
  - **營業稅** — 兩月一期申報（TET_U）
  - **報表** — 損益表 / 資產負債表，連續產出
  - **營所稅** — 年度申報（未來）
  - **節稅** — 即時提示（未來）

**關鍵原則：資料只收一次、跨模組重用。** 發票同時餵養營業稅、報表、營所稅、節稅。模組是同一份記帳核心的**消費端 / 視圖**，不是各自收料的孤島。這同時約束 IA 與架構：日後建營所稅 / 節稅時，要讀既有分錄 + 文件，**不**重新向客戶收料。

兩種模組性質，影響強調方式：

- **合規 / 義務**：營業稅、營所稅（週期性的必辦事項）
- **價值 / 成果**：報表、節稅（讓客戶感到「被服務」）

## 2. Portal 資訊架構（導覽）

以模組為頂層、可隨產品成長擴充：

```
總覽            ← dashboard：各模組狀態 + 待辦
營業稅          ← 依期別上傳發票 / 折讓 + 申報狀態（吸收現行「憑證上傳中心」）
報表            ← 損益表 / 資產負債表（本次新增）
其他文件        ← 非 VAT 的文件輸入（收據 / 帳單；餵養營所稅與存底）
[營所稅]        ← 未來
[節稅]          ← 未來
設定
```

兩個內建決策（待確認）：

1. **週期性發票 / 折讓上傳「歸在營業稅之下」**：期別 (YYYMM 兩月一期) 本就是營業稅概念；其他模組在背後重用同一份資料。客戶**每種資料只在一個地方上傳**，不會為了「報表」或「營所稅」重複上傳發票。這是最大的 UX 收益，也與架構（共用核心）一致。
2. **「其他文件」終於有清楚身分**：「不是 VAT 發票 / 折讓、但與你的存底 / 營所稅有關的文件」。一旦營業稅成為清楚命名的模組，其他文件就不再與之競爭。（營所稅上線後，其他文件可能改掛在其下、或維持為通用輸入區，屆時再議。）

## 3. 第一個實作切片：導覽改名（吸收自原 nav 筆記）

這是邁向模組 IA 的 step 1，獨立小 PR 即可交付。

**問題回顧**：首頁標題為 `<h1>憑證上傳中心</h1>`（`portal/page.tsx:160`）。「憑證」是廣義單據，讀起來像「所有文件都上傳這裡」，與另開的「其他文件」相牴觸。`首頁` 標籤也不暗示它其實是「依期別上傳統一發票 / 折讓」。**根因不是「其他文件」，而是營業稅流程被命名得太籠統（首頁）又太廣（憑證上傳中心）。**

**目前狀態**：
- 側欄 `components/portal-sidebar.tsx`：首頁、其他文件、設定。
- 底部導覽 `components/portal-bottom-nav.tsx`（mobile）：首頁、其他文件、設定、登出。
- 首頁 `app/firm/[firmId]/client/[clientId]/portal/page.tsx`：`<h1>憑證上傳中心</h1>` + 「本期優先處理」+「其他期別」；點期別 → portal 期別頁（4 張 type 卡 + FAB）上傳發票 / 折讓。

**改動（三件，且 copy 要等 nav 不再誤導才有效）**：

1. **改名營業稅流程（最高槓桿）**：`憑證上傳中心` → 模組名「營業稅」（內頁可保留副標如「依申報期別上傳統一發票與折讓單」）；nav `首頁` → 與目的地相符。「憑證」也是傳票 / 分錄模組用語，移除可消去第二層歧義。
2. **主動引導（交叉指引）**：每個上傳面放一行指向另一邊。期別頁「不是發票 / 折讓？請到『其他文件』」；`/documents`「統一發票與折讓單請到『營業稅』」。
3. **情境式 copy**：`/documents` 空狀態 + 上傳區 copy（描述放什麼、不放什麼）。**注意不過度警告**：其他文件要維持成好按的逃生口，避免客戶退回去把單據都當發票上傳（更糟的錯誤），只需溫和的「發票請到營業稅」指引。

**結構抉擇（已被模組視角推進）**：原 nav 筆記建議「就地改名」為 80/20。但在四模組的脈絡下，「首頁＝期別列表」不可擴充，因此方向改為：**首頁 → 真正的「總覽」**，期別流程移到「營業稅」模組之下。即先前的 hub 選項，現由模組結構（而非單純改名）所支撐。

**命名一致性**：選定的營業稅 / 各模組名稱，要在側欄、底部導覽、交叉指引、內頁標題**處處一致**。

## 4. 跨模組決策

- **「對客戶公開」閘門（per-module）**：報表是「我們服務的成果」，但通常要事務所複核 / 鎖定後才對客戶可見（不要露出記帳到一半的損益表）。每個模組應有「已對客戶發布」狀態。**這是真正擋住 IS/BS portal 頁開工的決策。**
- **各模組各有時間模型**：營業稅 = 期別（兩月一期）、報表 = 任意日期區間、營所稅 = 年度。導覽 / 落地頁要反映各自節奏；**不要**把營業稅的期別選擇器硬套到報表（報表沿用 firm 端的日期區間 / ROC 期別快捷）。
- **firm 端鏡像同一套模組命名**：firm 側欄客戶子模組已有傳票 / 損益表 / 資產負債表 / 其他文件；模組正名後，兩側用同一套詞彙，避免雙語彙。

## 5. 待確認決策

1. 確認 portal 採**模組化頂層導覽**方向（含 首頁 → 總覽）。
2. 「對客戶公開」閘門的形狀（per-module 狀態 + 由誰、何時切換）。
3. 各模組落地頁的時間模型（期別 / 日期區間 / 年度）落地細節。
4. 其他文件在營所稅上線後的歸屬（暫頂層）。

## 6. 與其他工作的關係

- **報表 (IS/BS) portal 頁**：在本 IA 的「報表」模組內設計與實作；開工前需先定「對客戶公開」閘門（決策 #2）。
- **Upload Classifier 計畫**（`UPLOAD_CLASSIFIER_PHASED_PLAN.md`）：其他文件 / `/documents` 屬第 1 層輸入；classifier 之客戶端曝光分階段（後續步驟 A/B）與本 IA 並行、互不阻擋。
